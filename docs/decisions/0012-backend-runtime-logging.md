# 0012 Backend Runtime Logging

## 状态

已采纳。

## 日期

2026-05-27，Asia/Shanghai。

## 背景

`v0.5.1` 规划新增 `task-service` 和 `notification-service` 后，项目将从单一业务后端进入多业务服务协作阶段。后续 `v0.6` Nacos、`v0.7` Redis、`v0.8` WebSocket、`v1.0` MQ、`v1.5` Seata 和 `v1.7` 可观测性都会依赖清晰的运行状态观察能力。

如果没有统一日志基线，学习时只能依赖接口返回结果判断系统是否正常，难以直观看到请求是否进入服务、服务间调用是否发生、缓存是否命中、消息是否发布、异常发生在哪一层。

因此在 `v0.5.1` 和 `v0.6` 之间增加 `v0.5.2 Backend Runtime Logging`，先建立控制台日志、文件日志和日志级别配置规则。

## 决策

采用 Spring Boot 默认日志体系作为本阶段日志方案：

| 项目 | 决策 |
|---|---|
| 日志门面 | SLF4J |
| 日志实现 | Logback |
| 控制台输出 | 必须支持 |
| 文件输出 | 必须支持 |
| 日志目录 | 项目运行目录下的 `logs` |
| 日志级别 | 通过配置文件、profile 或环境变量调整 |
| 日志内容 | 以运行排查和学习观察为目标，避免无意义刷屏 |
| 代码注释 | 类/方法注释与方法内部关键代码块注释并重 |
| 敏感信息 | 禁止打印密码、JWT 完整 token、Authorization header、数据库密码和真实密钥 |

## 服务文件约定

| 服务 | 日志文件 |
|---|---|
| `java-demo-app` | `logs/java-demo-app.log` |
| `task-service` | `logs/task-service.log` |
| `notification-service` | `logs/notification-service.log` |

如果后续容器化运行，容器内日志路径可以调整，但必须在 README 或部署文档中记录宿主机映射路径。

## 日志级别约定

| 级别 | 使用约定 |
|---|---|
| `DEBUG` | 本地排查、参数摘要、缓存命中、服务调用细节 |
| `INFO` | 服务启动、业务成功、关键状态变化 |
| `WARN` | 可预期异常、校验失败、登录失败、认证失败、可恢复下游失败 |
| `ERROR` | 未预期异常、服务不可用、配置错误、关键写入失败 |

默认使用 `INFO`。需要排查问题时，只调高局部包或单个服务的日志级别，不建议长期把 root 级别设置为 `DEBUG`。

## 代码注释约定

`v0.5.2` 修改的是后端运行基础能力，后续排查问题时会频繁阅读日志配置、请求上下文、认证、服务间调用和异常处理相关代码。因此本阶段以及后续后端开发必须遵守以下注释约定：

1. 类和公开方法使用中文 JavaDoc 说明职责、参数、返回值、异常语义和调用场景。
2. 方法内部关键流程、关键分支和关键代码块前必须增加中文注释。
3. requestId / MDC 生成、传递、清理逻辑必须说明为什么需要清理上下文。
4. JWT 解析、认证失败、异常转换、服务间调用、日志级别配置和日志文件路径处理必须说明设计意图。
5. 后续缓存、限流、MQ、WebSocket、分布式事务等后端能力也要在关键代码块前补充说明。
6. 注释解释“为什么这样做”和“当前版本边界”，不为普通赋值或显而易见的判断重复解释。

## 关键日志边界

后续所有后端 milestone 都应在以下位置补充必要日志：

1. 服务启动和关键配置加载。
2. 请求进入和请求完成。
3. 登录、JWT 签发、JWT 解析和认证失败。
4. 用户、任务、通知等核心业务状态变化。
5. 服务间调用开始、完成和失败。
6. 缓存命中、缓存失效、限流触发。
7. 消息发布、消息消费、重试和死信。
8. WebSocket 连接、断开、推送和失败。
9. 事务提交、回滚和补偿。
10. 全局异常处理和不可预期错误。

## 安全约束

日志中禁止出现以下内容：

1. 明文密码。
2. 密码哈希。
3. 完整 JWT。
4. `Authorization` header 完整值。
5. JWT secret。
6. 数据库密码。
7. 中间件真实账号密码和密钥。
8. 大段请求体或附件内容。

需要定位问题时，优先打印 userId、username、业务 ID、请求 ID、错误码、耗时和脱敏摘要。

## 与 v1.7 可观测性的关系

`v0.5.2` 只解决应用自身能不能打印、配置和保存日志。`v1.7 Observability` 再把日志接入 ELK，并与指标、链路追踪、trace id 关联。

也就是说：

| 版本 | 重点 |
|---|---|
| `v0.5.2` | 本地控制台日志、文件日志、日志级别配置、敏感信息保护 |
| `v1.7` | 日志采集、检索、分析、trace id 关联、跨服务可观测 |

## 影响

1. `v0.5.2` 成为进入 Nacos 前的后端日志基线版本。
2. 后续后端功能开发默认需要同步考虑关键日志点。
3. 后续后端功能开发默认需要同步补充类/方法注释和方法内部关键代码块注释。
4. 每个后端 milestone 的验收不只看接口是否成功，还要看控制台、日志文件和代码注释是否能帮助理解执行状态。
5. 日志内容必须服务于学习和排查，不能为了“有日志”而大量打印无业务含义的信息。
