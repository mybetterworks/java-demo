# 0015 Login Risk Slider Captcha

## 状态

已采纳。

## 日期

2026-05-31，Asia/Shanghai。

## 背景

当前项目已经具备账号密码登录、JWT 签发、Gateway JWT 校验、React/Vue 登录页和后端运行日志。随着项目逐步接近真实后台系统，需要在登录流程中加入最小风险控制，避免无限制密码尝试。

为了保持学习路径清晰，本阶段先实现一个轻量登录风险验证：

```text
5 分钟内登录失败 3 次
  -> 后续登录需要账号密码 + 滑块验证码
```

## 决策

在 `v0.5.4` 新增登录滑块验证码版本。

| 项目 | 决策 |
|---|---|
| 触发条件 | 5 分钟内登录失败 3 次 |
| 触发结果 | 后续登录必须完成滑块验证码 |
| 验证方式 | 账号密码 + 滑块验证码 |
| 前端范围 | React 和 Vue 登录页都要同步支持 |
| 存储方式 | MVP 可用单机内存或轻量本地存储，`v0.7` 迁移 Redis |
| 安全边界 | 不打印密码、完整 JWT、验证码答案或真实密钥 |

## 统计维度

MVP 推荐使用：

```text
username + clientIp
```

原因：

1. 只按 username 统计，攻击者可能让某个账号长期进入验证码状态。
2. 只按 IP 统计，多个用户共用网络时可能互相影响。
3. `username + clientIp` 对学习型项目足够直观，也便于后续迁移 Redis key。

如果实现时为了简化选择其他维度，必须在 `docs/PROGRESS.md` 中记录原因和风险。

## 状态生命周期

| 状态 | 生命周期 |
|---|---|
| 登录失败计数 | 5 分钟 TTL |
| 验证码 challenge | 建议 2 分钟 TTL |
| 验证通过 token | 短 TTL，一次性使用 |
| 登录成功 | 清理失败计数和验证码状态 |

## 与 Redis 的关系

`v0.5.4` 不强制引入 Redis，避免在登录安全版本中同时引入新中间件。

`v0.7 Redis` 再把以下状态迁移到 Redis：

1. 登录失败计数。
2. 验证码 challenge。
3. 验证通过 token。
4. 必要的限流状态。

## 影响

1. 登录流程从单纯账号密码变为“低风险账号密码，高风险账号密码 + 滑块验证码”。
2. React 和 Vue 登录页必须同步支持验证码交互，不能只改后端。
3. 后端日志需要能观察失败计数和验证码触发，但不能泄露敏感信息。
4. 后续 Redis、MQ、可观测性版本可以基于该能力继续扩展安全审计和风险分析。
