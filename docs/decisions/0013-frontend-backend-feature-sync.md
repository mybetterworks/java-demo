# 0013 Frontend Backend Feature Sync

## 状态

已采纳。

## 日期

2026-05-27，Asia/Shanghai。

## 背景

项目已经同时拥有 React 管理端和 Vue 管理端，并在 `v0.5.1` 开始新增 `task-service` 和 `notification-service`。如果后续只开发后端接口，而不及时判断前端是否需要同步，项目会出现“接口可用但用户无法在页面操作”的断层。

本项目的目标是面向 Java 微服务和全栈开发面试学习，因此后端能力、前端页面、接口联调和用户可见体验需要一起演进。与此同时，React 和 Vue 两套前端承担对比学习价值，不能为了省事只改其中一套。

## 决策

从 `v0.5.3` 开始，后续进行后台代码开发时，必须自动判断是否需要同步新增或修改前端页面功能。

核心规则：

| 规则 | 说明 |
|---|---|
| 后端新增用户可见能力 | 默认评估 React 和 Vue 是否需要新增页面、入口、按钮、筛选项或状态展示 |
| 后端修改接口契约 | 默认同步检查前端 API 封装、类型定义、字段渲染和错误处理 |
| 后端新增业务状态 | 默认同步检查前端标签、筛选项、表单选项和空状态 |
| 后端新增错误码或限制 | 默认同步检查前端提示文案和异常处理 |
| 内部透明优化 | 如日志、缓存透明优化、配置中心迁移，通常只需要前端回归验证，不强制新增页面 |
| 用户可感知中间件能力 | 如 WebSocket、文件上传、搜索、统计看板，必须同步设计 React 和 Vue 前端能力 |

除非用户明确要求“只做后端”“只做文档”或“不要修改前端”，否则 Codex 在实现后端 milestone 时应主动完成必要的前端联动修改。

## 双前端一致性

React 和 Vue 的业务表现必须保持一致：

1. 菜单入口一致。
2. 页面名称一致。
3. 操作路径一致。
4. 表格字段和筛选项一致。
5. 表单字段和校验语义一致。
6. 错误提示和空状态语义一致。
7. 登录态、无权限、接口失败的用户反馈一致。

但代码实现不要求一致：

| 前端 | 保留特点 |
|---|---|
| React | TypeScript、Ant Design、组件化页面、明确类型定义 |
| Vue | JavaScript、Element Plus、`views`、`layouts`、`composables`、`api`、`storage` 分层 |

## 注释要求

后续新增或修改前端代码时，必须补充详细中文注释，尤其是：

1. API 封装和请求参数。
2. 页面状态和加载流程。
3. 表单校验和提交逻辑。
4. 表格分页、筛选和刷新逻辑。
5. 登录态恢复和 token 使用。
6. 本地缓存策略。
7. React 与 Vue 实现差异背后的框架原因。

注释目标是帮助学习者阅读和复盘，不是机械重复代码。

## 判断清单

每次后端开发完成或接口发生变化时，至少检查：

| 问题 | 如果答案是“是” |
|---|---|
| 用户是否需要在页面触发这个能力？ | 增加或修改 React/Vue 页面入口和操作按钮 |
| 用户是否需要看到新的字段？ | 增加或修改 React/Vue 表格列、详情展示或表单项 |
| 是否新增业务状态？ | 增加或修改状态标签、筛选项和状态流转按钮 |
| 是否新增错误场景？ | 增加或修改前端错误提示和兜底处理 |
| 是否改变分页、排序或查询参数？ | 修改 React/Vue 查询条件、缓存和 API 封装 |
| 是否只是内部透明优化？ | 记录无需新增页面的原因，并做必要前端回归 |

## 影响

1. `v0.5.3` 成为任务和通知服务的前端承接版本。
2. 后续后端 milestone 不再只看后端接口是否完成，也要评估用户是否需要页面入口。
3. React 和 Vue 双端一致性成为长期规则，而不是一次性要求。
4. 如果某次后端能力暂不做前端，需要在 `docs/PROGRESS.md` 或对应 milestone 中说明原因。
