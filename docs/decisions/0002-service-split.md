# 0002 Service Split Strategy

状态：已采纳。

日期：2026-05-20。

## 背景

本项目的学习目标覆盖完整微服务技术栈，但业务范围很小，只有登录和用户管理。如果一开始就拆成大量微服务，会让注册中心、网关、配置、链路、事务、部署问题同时出现，反而降低学习效率。

## 决策

采用“先单体闭环，再模块化，再微服务拆分”的路线。

阶段安排：

| 阶段 | 服务形态 | 说明 |
|---|---|---|
| `v0.1` - `v0.4` | 单体或简单多模块 | 先跑通登录、用户管理、前端访问 |
| `v0.5` | 引入网关 | 外部流量先进入 Gateway |
| `v0.5.1` | 新增轻量业务微服务 | 增加任务服务和通知服务，先用静态 REST 调用形成跨服务业务闭环 |
| `v0.5.2` | 建立后端日志基线 | 用户、任务、通知三个业务服务具备控制台日志、文件日志和日志级别配置 |
| `v0.5.3` | 补齐任务和通知前端 | React 和 Vue 都承接任务/通知微服务能力，形成浏览器端业务闭环 |
| `v0.6` | 引入注册中心 | Gateway、用户、任务、通知服务接入 Nacos |
| `v1.5` 前后 | 强化分布式一致性验证 | 在已有用户、任务、通知服务边界上验证 Seata 和链路追踪 |

## 初始服务边界建议

| 服务 | 职责 |
|---|---|
| `gateway` | 路由、JWT 校验、统一入口、限流 |
| `auth-service` | 登录、发 token、刷新 token、退出登录 |
| `user-service` | 用户资料、角色字段、用户管理 |
| `task-service` | 任务创建、分配、状态流转、任务查询 |
| `notification-service` | 站内信、未读数、已读状态、后续 WebSocket 推送源 |
| `common` | 统一返回、异常、工具、公共 DTO |

早期可以先把 `auth` 和 `user` 放在一个 Spring Boot 应用中，等网关和 Nacos 阶段再拆。

## 原则

1. 先让业务闭环，再拆服务。
2. 拆分必须带来学习价值或技术验证价值。
3. 不为了“看起来像微服务”提前制造分布式复杂度。
4. 每次拆分后都必须保留可运行版本。

## 影响

1. `v0.1` 可以使用单应用降低启动成本。
2. `v0.5` 开始外部入口统一改为 Gateway。
3. `v0.5.1` 开始拥有用户、任务、通知三个业务域，后续中间件实验应优先围绕这些业务域设计。
4. `v0.5.2` 建立后端运行日志基线，后续服务治理、缓存、消息和可观测性实验都应保留关键日志。
5. `v0.5.3` 让任务和通知能力进入 React/Vue 页面，后续用户可见后端能力变化需要同步评估前端。
6. `v0.6` 后服务发现和配置管理进入主线。
7. `v1.5` 的 Seata 验证可以直接基于任务和通知跨服务链路设计，不需要临时补造业务服务。
