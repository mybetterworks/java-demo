# v0.5.1 Task And Notification Services

## 目标

在 `v0.5 Gateway JWT` 已完成统一入口的基础上，新增两个最小 MVP 微服务：

| 服务 | 目标 |
|---|---|
| `task-service` | 提供任务创建、分配、查询、状态流转和逻辑删除能力 |
| `notification-service` | 提供站内信创建、查询、未读数量和已读标记能力 |

本版本的重点不是一次性引入 Nacos、Redis 或 MQ，而是先补齐真实业务服务边界，让后续 `v0.6` 之后的注册发现、缓存、消息、推送、搜索、事务、链路追踪和部署实验有更自然的业务承载。

## 新增能力

最小多微服务业务闭环。

```text
React / Vue
  -> Gateway
    -> java-demo-app
    -> task-service
    -> notification-service
```

| 类别 | 技术 |

## 服务职责

| 服务 | 职责 | 本版本边界 |
|---|---|---|
| `java-demo-app` | 用户、登录、JWT、用户管理 | 继续作为认证和用户资料来源 |
| `task-service` | 任务和任务状态 | 依赖当前登录用户，必要时调用 `java-demo-app` 校验被分配用户 |
| `notification-service` | 站内信和未读数 | 被 `java-demo-app` 或 `task-service` 调用创建通知 |
| `gateway` | 统一入口和 JWT 基础校验 | 新增 `/api/tasks/**` 和 `/api/notifications/**` 路由 |

## task-service MVP

### 功能范围

| 功能 | 说明 |
|---|---|
| 创建任务 | 登录用户创建任务，可指定负责人 |
| 我的任务 | 查询当前登录用户创建或负责的任务 |
| 任务分页 | 管理端分页查询任务 |
| 任务详情 | 根据 ID 查询任务 |
| 状态流转 | 修改任务状态 |
| 逻辑删除 | 删除任务时保留数据，便于后续审计和搜索 |

### 建议接口

| 方法 | 路径 | 说明 | JWT |
|---|---|---|---|
| `GET` | `/api/tasks/health` | 任务服务健康检查 | 否 |
| `POST` | `/api/tasks` | 创建任务 | 是 |
| `GET` | `/api/tasks/my` | 查询我的任务 | 是 |
| `GET` | `/api/tasks` | 分页查询任务 | 是 |
| `GET` | `/api/tasks/{id}` | 查询任务详情 | 是 |
| `PUT` | `/api/tasks/{id}` | 更新标题、描述、负责人、优先级、截止时间 | 是 |
| `PUT` | `/api/tasks/{id}/status` | 修改任务状态 | 是 |
| `DELETE` | `/api/tasks/{id}` | 逻辑删除任务 | 是 |

### 建议数据模型

| 字段 | 说明 |
|---|---|
| `id` | 主键 |
| `title` | 任务标题 |
| `description` | 任务描述 |
| `creator_user_id` | 创建人用户 ID |
| `assignee_user_id` | 负责人用户 ID |
| `status` | `TODO`、`IN_PROGRESS`、`DONE`、`CANCELLED` |
| `priority` | `LOW`、`MEDIUM`、`HIGH` |
| `due_time` | 截止时间 |

## 服务间调用

本版本先使用最小 REST 调用，避免在 `v0.5.1` 同时引入 Nacos、OpenFeign、MQ 等新复杂度。

| 调用方 | 被调用方 | 场景 |
|---|---|---|
| `task-service` | `java-demo-app` | 创建或更新任务时校验 `assigneeUserId` 是否存在 |
| `task-service` | `notification-service` | 任务分配或状态变化后创建通知 |
| `java-demo-app` | `notification-service` | 用户注册或管理员创建用户后创建欢迎通知，可作为可选验收 |

调用方式建议：

| 阶段 | 方式 |
|---|---|
| `v0.5.1` | 静态地址 REST 调用，例如 `http://localhost:8091`、`http://localhost:8094` |
| `v0.6` | 迁移为 Nacos 服务发现调用 |
| `v1.0` | 任务事件和通知创建从同步 REST 演进为 MQ 异步事件 |

## 开发步骤

1. 在 `backend` 下新增 `task-service` 和 `notification-service` Maven 模块，参照 `java-demo-app` 的包结构、统一响应、异常处理、JWT 解析、OpenAPI 配置和测试风格。
2. 为两个服务分别配置端口、应用名、数据源、MyBatis Plus、SQL 初始化和本地 profile。
3. 为 MySQL 单实例新增 `java_demo_task` 和 `java_demo_notification` database 初始化说明或脚本。
4. 实现 `task-service` 最小任务 CRUD 和状态流转接口。
5. 实现 `notification-service` 最小通知创建、查询、未读数和已读接口。
6. 在 `task-service` 中通过 REST 调用 `java-demo-app` 校验被分配用户。
7. 在 `task-service` 中通过 REST 调用 `notification-service` 创建任务通知。
8. 在 Gateway 中新增任务和通知服务静态路由，并调整 JWT 白名单。
9. React 和 Vue 前端可选增加任务和通知入口；如果本版本控制范围需要更小，可先只用 Swagger 或接口脚本验收。
10. 补充自动化测试和最小真实联调验证。
11. 更新 README、`docs/PROGRESS.md` 和必要决策文档。

## 验收标准

1. `java-demo-app`、`gateway`、`task-service`、`notification-service` 都可以启动。
2. Gateway 可以访问三个业务服务的健康检查。
3. 不带 JWT 访问任务和通知业务接口会被 Gateway 拦截。
4. 携带有效 JWT 可以创建任务、查询我的任务、修改任务状态。
5. 创建或分配任务后，`notification-service` 可以生成对应通知。
6. 当前用户可以查询自己的通知和未读数量，并可标记已读。
7. `task-service` 创建任务时可以校验被分配用户是否存在。
8. 现有登录、用户管理、React、Vue 功能不回归。
9. OpenAPI 或接口文档能覆盖新增服务接口。
10. 新增服务端口、数据库、启动顺序、验证命令已记录。
