# v0.5 Gateway JWT

## 目标

引入 Spring Cloud Gateway，所有前端和外部请求统一经过网关，由网关完成基础 JWT 校验和路由转发。

## 新增能力

统一网关入口。

## 技术范围

| 类别 | 技术 |
|---|---|
| 网关 | Spring Cloud Gateway |
| 认证 | JWT 校验过滤器 |
| 后端 | 原有登录和用户服务 |

## 架构变化

```text
Browser
  -> Gateway
    -> Backend App or Auth/User Services
```

早期可以先让 Gateway 转发到单个后端应用。服务拆分可以在后续 Nacos 阶段继续推进。

## 路由建议

| 外部路径 | 目标 |
|---|---|
| `/api/auth/**` | 登录认证服务或后端应用 |
| `/api/users/**` | 用户服务或后端应用 |
| `/api/health` | 健康检查 |

## 开发步骤

1. 调整后端为 Maven 多模块或新增 `gateway` 模块。
2. 创建 Gateway 应用。
3. 配置路由规则。
4. 实现 JWT 校验过滤器。
5. 配置登录、注册、健康检查等白名单。
6. 统一 CORS 处理。
7. 修改 React 和 Vue 前端 API 地址，指向 Gateway。
8. 验证直连后端和经过网关的行为差异。
9. 更新 README 和 `docs/PROGRESS.md`。

## 验收标准

1. Gateway 可以启动。
2. 登录请求可以不带 token 通过 Gateway。
3. 用户管理请求不带 token 会被 Gateway 拦截。
4. 用户管理请求携带有效 token 可以通过 Gateway。
5. React 和 Vue 前端都通过 Gateway 调用 API。
6. 后端直接访问方式在文档中明确为开发调试用途。

## 完成定义

1. 所有外部 API 统一走 Gateway。
2. JWT 网关校验可验证。
3. 前端联调通过。
4. 新增和修改代码已补充详细中文注释。
5. 如果需要保存版本，由用户手动提交并手动打 tag：`v0.5-gateway-jwt`；Codex 不自动执行提交、tag 或 push。

## 实现记录

完成时间：2026-05-26，Asia/Shanghai。

本次实现保持“先单体闭环，再逐步微服务化”的路线：业务后端仍运行在 `backend/app`，新增 `backend/gateway` 作为统一外部入口。Gateway 当前使用静态后端地址 `http://localhost:8091` 转发请求，后续 `v0.6 Nacos` 再演进为服务发现路由。

| 项目 | 实现 |
|---|---|
| Maven 结构 | 根 `pom.xml` 新增 `backend/gateway` 模块，项目版本升级为 `0.5.0-SNAPSHOT` |
| Spring Cloud | 引入 Spring Cloud `2023.0.3` 和 `spring-cloud-starter-gateway` |
| 网关端口 | `8092`，避开本机占用范围 `7991-8090`、`8146-8245` |
| 后端端口 | `8091`，继续作为后端服务端口和开发调试直连入口 |
| 路由 | `/api/**`、`/v3/api-docs/**`、`/swagger-ui.html`、`/swagger-ui/**` 转发到后端 |
| JWT 校验 | `JwtGatewayFilter` 在网关层校验受保护 `/api/**` 请求的 Bearer token |
| 白名单 | `/api/health`、`/api/auth/register`、`/api/auth/login`、OpenAPI 和 Swagger UI 不要求 token |
| 前端代理 | React 和 Vue 的 Vite proxy 均从后端 `8091` 调整为 Gateway `8092` |
| 防御边界 | 后端 `AuthInterceptor` 保留，形成“网关校验 + 后端校验”的双层保护 |
