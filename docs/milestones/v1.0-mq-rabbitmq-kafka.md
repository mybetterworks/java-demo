# v1.0 RabbitMQ And Kafka

## 目标

引入消息队列，把登录审计、通知或用户变更事件从同步流程中拆出来。RabbitMQ 作为主线实现，Kafka 作为事件流对照学习。

## 新增能力

异步消息。

## 技术范围

| 类别 | 技术 |
|---|---|
| 消息队列主线 | RabbitMQ |
| 事件流对照 | Kafka |
| 业务场景 | 登录审计、通知、用户变更 |

## 功能范围

| 功能 | 说明 |
|---|---|
| 登录事件发布 | 用户登录成功后发布事件 |
| 审计消费者 | 消费事件并写入审计表 |
| 通知事件 | 用户创建或状态变化后发送通知 |
| Kafka 对照 | 使用相同事件模型做最小验证 |

## 容器策略

遵守 `docs/decisions/0010-docker-service-containerization.md`。

| 服务 | 容器要求 |
|---|---|
| RabbitMQ | 单节点使用 `java-demo-rabbitmq-1` 独立容器；集群时每个 RabbitMQ broker 独立容器 |
| Kafka | 单节点使用 `java-demo-kafka-1` 独立容器；集群时每个 broker/controller 独立容器 |
| 辅助组件 | 如果 Kafka 方案需要额外协调或控制组件，该组件也必须独立容器，不与 Kafka broker 合并 |
| 对比环境 | RabbitMQ 和 Kafka compose 分开维护，便于单独启动、停止和对照验证 |

## 开发步骤

1. 新增 RabbitMQ Docker Compose，记录镜像版本、容器名、端口、volume、network 和健康检查。
2. 定义事件消息模型。
3. 登录成功后发布登录审计事件。
4. 实现 RabbitMQ 消费者并写入审计表。
5. 实现消息失败重试或死信队列基础配置。
6. 新增 Kafka Docker Compose 或对照环境，确保 Kafka 与其辅助组件容器边界清晰。
7. 使用 Kafka 实现一个最小事件发布和消费示例。
8. 记录 RabbitMQ 与 Kafka 在本项目场景中的区别。
9. 更新 README 和 `docs/PROGRESS.md`。

## 验收标准

1. RabbitMQ 可以启动并访问管理控制台。
2. 登录成功后可以产生消息。
3. 消费者可以消费消息并写入审计记录。
4. 消费失败有可观察的处理方式。
5. Kafka 最小发布消费链路可以运行。
6. 同步登录链路不因审计写入失败而失败。
7. RabbitMQ、Kafka 及其集群扩展容器边界已记录。
