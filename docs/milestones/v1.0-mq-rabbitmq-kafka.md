# v1.0 RabbitMQ And Kafka

## 目标

引入消息队列，把登录审计、任务事件、通知创建或用户变更事件从同步流程中拆出来。RabbitMQ 作为主线实现，Kafka 作为事件流对照学习。

## 新增能力

异步消息。

## 技术范围

| 类别 | 技术 |
|---|---|
| 消息队列主线 | RabbitMQ |
| 事件流对照 | Kafka |
| 业务场景 | 登录审计、任务事件、通知、用户变更 |

## 功能范围

| 功能 | 说明 |
|---|---|
| 登录事件发布 | 用户登录成功后发布事件 |
| 审计消费者 | 消费事件并写入审计表 |
| 任务事件发布 | 任务创建、分配、状态变化后发布事件 |
| 通知事件消费 | 通知服务消费任务事件和用户事件后创建站内信，替代 `v0.6.1` 中的同步 Feign 通知调用 |
| 通知事件 | 用户创建或状态变化后发送通知 |
| Kafka 对照 | 使用相同事件模型做最小验证 |

## 日志要求

继承 `v0.5.2` 后端运行日志基线。本阶段需要重点记录消息发布、消息消费、消费成功、消费失败、重试、死信、消息幂等和消息 ID，便于观察同步 REST 链路演进为异步事件链路后的执行状态。

## 容器策略

遵守 `docs/decisions/0010-docker-service-containerization.md`。

| 服务 | 容器要求 |
|---|---|
| RabbitMQ | 单节点使用 `java-demo-rabbitmq-1` 独立容器；集群时每个 RabbitMQ broker 独立容器 |
| Kafka | 单节点使用 `java-demo-kafka-1` 独立容器；集群时每个 broker/controller 独立容器 |
| 辅助组件 | 如果 Kafka 方案需要额外协调或控制组件，该组件也必须独立容器，不与 Kafka broker 合并 |
| 对比环境 | RabbitMQ 和 Kafka compose 分开维护，便于单独启动、停止和对照验证 |

## 开发步骤

1. 新增 RabbitMQ Docker Compose，记录镜像版本、容器名、端口、volume、network 和健康检查。
2. 定义事件消息模型。
3. 登录成功后发布登录审计事件。
4. 任务创建、任务分配或任务状态变化后发布任务事件。
5. `notification-service` 消费任务事件并创建站内信，替代 `v0.6.1` 中的同步 Feign 通知调用。
6. 实现 RabbitMQ 消费者并写入审计表。
7. 实现消息失败重试或死信队列基础配置。
8. 新增 Kafka Docker Compose 或对照环境，确保 Kafka 与其辅助组件容器边界清晰。
9. 使用 Kafka 实现一个最小事件发布和消费示例。
10. 记录 RabbitMQ 与 Kafka 在本项目场景中的区别。
11. 在发布、消费、重试、死信和幂等处理处补充必要日志。
12. 更新 README 和 `docs/PROGRESS.md`。

## 验收标准

1. RabbitMQ 可以启动并访问管理控制台。
2. 登录成功后可以产生消息。
3. 消费者可以消费消息并写入审计记录。
4. 任务事件可以被通知服务消费并生成通知。
5. 消费失败有可观察的处理方式。
6. Kafka 最小发布消费链路可以运行。
7. 同步登录和任务创建链路不因审计或通知写入失败而失败。
8. 控制台和文件日志可以观察消息发布、消费、失败重试和死信处理。
9. RabbitMQ、Kafka 及其集群扩展容器边界已记录。
