# v0.5.3 Task And Notification Frontends

## 目标

在 `v0.5.1 Task And Notification Services` 完成任务服务和通知服务 MVP、`v0.5.2 Backend Runtime Logging` 建立后端日志基线后，为 React 和 Vue 两套前端补齐任务与通知相关页面。

本版本的重点是让前端真正承接 `task-service` 和 `notification-service` 的业务能力，让用户不只通过 Swagger 或接口脚本验证任务和通知，而是可以在浏览器中完成任务创建、查询、状态流转、通知查看和已读操作。

## 新增能力

React 和 Vue 双前端任务/通知管理能力。

```text
React 管理端
  -> 任务管理
  -> 通知中心

Vue 管理端
  -> 任务管理
  -> 通知中心
```

两套前端需要保持业务功能、页面布局、操作路径和接口行为一致，但代码结构、语言和开发风格继续保留各自特点：

| 前端 | 技术风格 |
|---|---|
| `frontend-react` | React + TypeScript + Ant Design，接口类型、组件 props 和页面状态尽量类型明确 |
| `frontend-vue` | Vue 3 + JavaScript + Element Plus，使用 `views`、`layouts`、`composables`、`api`、`storage` 等 Vue 常见分层 |

## 技术范围

| 类别 | React | Vue |
|---|---|---|
| 语言 | TypeScript | JavaScript |
| 构建 | Vite | Vite |
| 组件库 | Ant Design | Element Plus |
| HTTP | 复用现有 API 封装 | 复用现有 API 封装 |
| 登录态 | IndexedDB 中的 `auth_session` | localStorage 中的登录会话 |
| 最近查询 | IndexedDB | localStorage |
| 代理 | Vite proxy 到 Gateway `8092` | Vite proxy 到 Gateway `8092` |

## 功能范围

### 导航和布局

| 功能 | 说明 |
|---|---|
| 侧边菜单 | 在现有管理端菜单中新增“任务管理”和“通知中心” |
| 首页入口 | 首页可展示任务和通知能力入口，是否做统计卡片可按实现复杂度控制 |
| 操作路径一致 | React 和 Vue 中菜单名称、页面结构、按钮语义尽量一致 |
| 登录态复用 | 继续使用现有 JWT 登录态，不新增独立认证逻辑 |

### 任务管理

| 功能 | 说明 |
|---|---|
| 我的任务 | 查询当前登录用户创建或负责的任务 |
| 任务分页 | 查询任务列表，支持分页和基础筛选 |
| 任务详情 | 查看任务标题、描述、状态、优先级、负责人和截止时间 |
| 创建任务 | 通过表单创建任务，可指定负责人、优先级和截止时间 |
| 编辑任务 | 修改标题、描述、负责人、优先级和截止时间 |
| 状态流转 | 支持 `TODO`、`IN_PROGRESS`、`DONE`、`CANCELLED` 等状态切换 |
| 逻辑删除 | 调用后端逻辑删除接口，并刷新列表 |

### 通知中心

| 功能 | 说明 |
|---|---|
| 我的通知 | 查询当前登录用户收到的通知 |
| 未读数量 | 展示当前用户未读通知数 |
| 通知列表 | 展示标题、内容摘要、类型、业务 ID、已读状态和创建时间 |
| 单条已读 | 标记指定通知为已读 |
| 全部已读 | 一键标记当前用户全部通知为已读 |
| 状态反馈 | 已读/未读用标签、颜色或状态列直观区分 |

## 建议页面结构

### React

React 端可以延续当前 `frontend-react` 的组织方式，不强行迁移到 Vue 风格目录。

建议新增或调整：

| 类型 | 建议 |
|---|---|
| API 封装 | 增加 `task` 和 `notification` API 方法，定义清晰 TypeScript 类型 |
| 页面组件 | 新增任务管理组件和通知中心组件 |
| 表单组件 | 任务新增/编辑表单可以用 Ant Design `Modal` + `Form` |
| 表格组件 | 使用 Ant Design `Table`、`Tag`、`Button`、`Popconfirm`、`Pagination` |
| 本地缓存 | 可将任务和通知查询条件保存到 IndexedDB，便于和现有用户查询缓存保持学习连续性 |

### Vue

Vue 端继续保持 Vue 常见分层，不要求和 React 文件结构一致。

建议新增或调整：

| 类型 | 建议 |
|---|---|
| `views` | 新增任务管理页和通知中心页 |
| `api` | 增加任务和通知 API 封装 |
| `composables` | 增加任务管理和通知中心组合式逻辑 |
| `storage` | 保存任务和通知最近查询条件 |
| UI 组件 | 使用 Element Plus 的 `el-table`、`el-form`、`el-dialog`、`el-tag`、`el-button`、`el-pagination` |

## 接口范围

本版本优先复用 `v0.5.1` 已规划的 Gateway 入口。

| 方法 | 路径 | 前端用途 |
|---|---|---|
| `GET` | `/api/tasks/my` | 查询我的任务 |
| `GET` | `/api/tasks` | 任务分页查询 |
| `GET` | `/api/tasks/{id}` | 任务详情 |
| `POST` | `/api/tasks` | 创建任务 |
| `PUT` | `/api/tasks/{id}` | 编辑任务 |
| `PUT` | `/api/tasks/{id}/status` | 修改任务状态 |
| `DELETE` | `/api/tasks/{id}` | 删除任务 |
| `GET` | `/api/notifications/my` | 查询我的通知 |
| `GET` | `/api/notifications/my/unread-count` | 查询未读数量 |
| `PUT` | `/api/notifications/{id}/read` | 单条已读 |
| `PUT` | `/api/notifications/read-all` | 全部已读 |

如果任务负责人选择需要用户列表，优先复用现有 `/api/users` 分页接口。MVP 阶段也可以先用负责人用户 ID 输入框，但如果实现成本可控，建议做用户选择器，学习体验会更贴近真实后台系统。

## 双前端一致性要求

| 维度 | 要求 |
|---|---|
| 菜单名称 | React 和 Vue 都使用“任务管理”“通知中心” |
| 页面信息架构 | 筛选区、表格区、分页区、操作按钮位置尽量一致 |
| 功能能力 | 不允许 React 有任务创建而 Vue 没有，或 Vue 有通知已读而 React 没有 |
| 接口语义 | 同一操作调用同一后端接口 |
| 错误提示 | 登录过期、无权限、接口失败、表单校验失败都要有用户可见提示 |
| 空状态 | 无任务、无通知时要有明确空状态 |
| 加载状态 | 查询、提交、删除、标记已读时要有 loading 或禁用按钮反馈 |
| 术语 | 状态、优先级、通知类型的中文展示尽量一致 |

## 代码注释要求

本版本是前端学习重点，新增和修改的前端代码必须补充详细中文注释。

| 位置 | 注释重点 |
|---|---|
| API 封装 | 说明接口用途、请求参数、返回数据结构和 JWT 复用方式 |
| 页面组件 | 说明页面职责、核心状态、加载流程和操作入口 |
| 表单逻辑 | 说明字段含义、校验规则、创建/编辑复用逻辑 |
| 表格逻辑 | 说明分页、筛选、刷新和状态展示逻辑 |
| 本地缓存 | 说明为什么 React 用 IndexedDB、Vue 用 localStorage，以及保存哪些查询条件 |
| 错误处理 | 说明接口失败、登录失效和业务错误如何反馈给用户 |
| 双端差异 | 如果 React 和 Vue 实现方式不同，需要用注释或文档说明差异来自框架习惯，而不是业务语义差异 |

注释要帮助学习“为什么这样做”，不要只重复代码字面含义。

## 后端开发联动前端规则

从本版本开始，后续进行后台代码开发时，只要出现面向用户的新接口、新业务字段、新业务状态或接口语义变化，就必须自动判断是否需要同步前端：

| 后端变化 | 前端处理 |
|---|---|
| 新增用户可操作接口 | 默认需要评估并补齐 React/Vue 页面入口 |
| 修改接口字段或响应结构 | 默认需要同步修改 React/Vue API 封装、类型或数据渲染 |
| 新增业务状态 | 默认需要同步修改状态标签、筛选项、表单选项 |
| 新增错误码或限流规则 | 默认需要同步修改前端错误提示 |
| 纯内部优化 | 如日志、缓存透明优化、配置中心迁移，通常只做回归验证，不强制新增页面 |
| 中间件带来用户可见能力 | 如 WebSocket 推送、文件上传、搜索、统计看板，需要同步开发前端 |

除非用户明确要求“只做后端”或“只做文档”，否则后端可见能力变化应在同一 milestone 中自动评估并补齐必要前端改动。

## 开发步骤

1. 阅读 `docs/ROADMAP.md`、`docs/DEVELOPMENT_RULES.md`、`docs/PROGRESS.md`、本 milestone 文档，以及 `v0.5.1` 任务/通知接口文档。
2. 确认 `v0.5.1` 和 `v0.5.2` 已完成必要验证，任务和通知接口可经 Gateway 访问。
3. 设计 React 与 Vue 一致的信息架构，包括菜单、任务页面、通知页面、筛选区、表格区和操作按钮。
4. 在 React 中补齐任务和通知 API 封装、TypeScript 类型、页面组件、表单弹窗、表格操作和 IndexedDB 查询条件缓存。
5. 在 Vue 中补齐任务和通知 API 封装、`views`、`composables`、Element Plus 表单/表格/弹窗和 localStorage 查询条件缓存。
6. 为新增和修改的前端代码补充详细中文注释。
7. 验证 React 和 Vue 都可以登录、进入任务管理、创建任务、修改任务状态、查看通知、标记已读。
8. 执行 React 和 Vue 的构建验证。
9. 使用真实 Gateway、任务服务、通知服务完成浏览器联调。
10. 更新 README 和 `docs/PROGRESS.md`，记录页面能力、启动方式、验收结果和已知问题。

## 验收标准

1. React 管理端新增“任务管理”和“通知中心”入口。
2. Vue 管理端新增“任务管理”和“通知中心”入口。
3. React 和 Vue 的任务页面功能一致，可以查询、创建、编辑、改状态和删除任务。
4. React 和 Vue 的通知页面功能一致，可以查询通知、查看未读数、单条已读和全部已读。
5. 两套前端都通过 Gateway `8092` 访问任务和通知接口。
6. 两套前端都能正确处理未登录、登录过期、接口失败和空数据。
7. 两套前端布局和操作路径保持一致，但代码结构保留各自框架特点。
8. 新增和修改的前端代码已补充详细中文注释。
9. React 构建通过。
10. Vue 构建通过。
11. README 和 `docs/PROGRESS.md` 已记录本版本能力和验证结果。

## 完成定义

1. React 和 Vue 都能在浏览器中完成任务/通知最小业务闭环。
2. 双端功能、布局、术语和操作路径一致。
3. React 保持 TypeScript + Ant Design 风格，Vue 保持 JavaScript + Element Plus + Vue 分层风格。
4. 新增前端代码具备详细中文注释，便于学习阅读。
5. 构建和真实联调验证已完成，或明确记录无法验证原因。
6. 后续后端开发联动前端的判断规则已同步到开发规则文档。
7. 如果需要保存版本，由用户手动提交并手动打 tag：`v0.5.3-task-notification-frontends`；Codex 不自动执行提交、tag 或 push。
