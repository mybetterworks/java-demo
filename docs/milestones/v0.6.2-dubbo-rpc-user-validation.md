# v0.6.2 Dubbo RPC User Validation

## 目标

在 `v0.6.1 OpenFeign Service Calls` 已完成声明式 HTTP 调用后，引入 Dubbo RPC，选择一条稳定、强类型、内部同步调用链路进行 RPC 学习和对比：

```text
task-service -> java-demo-app 用户校验
```

本版本不把所有服务间调用都改成 Dubbo，也不把 `task-service -> notification-service` 改成 Dubbo。通知创建链路短期保留 Feign，长期在 `v1.0 MQ` 演进为异步事件。

## 新增能力

内部 RPC 服务调用。

```text
task-service
  -> Dubbo RPC -> java-demo-app 用户校验

task-service
  -> OpenFeign -> notification-service 通知创建
```

## 改造范围

| 调用方 | 被调用方 | 业务目的 | v0.6.2 改造 |
|---|---|---|---|
| `task-service` | `java-demo-app` | 创建或更新任务时校验负责人用户是否存在 | 引入 Dubbo RPC |
| `task-service` | `notification-service` | 任务创建、分配或状态变化后创建通知 | 保留 OpenFeign，等待 `v1.0 MQ` 异步化 |

## 为什么只改用户校验链路

| 链路 | 推荐方式 | 原因 |
|---|---|---|
| `task-service -> java-demo-app` 用户校验 | Dubbo RPC | 同步、内部、强类型、接口稳定，适合学习 RPC |
| `task-service -> notification-service` 通知创建 | Feign 短期，MQ 长期 | 通知属于副作用链路，不适合长期同步 RPC |
| 前端 / Gateway / Swagger | REST | 对外 API 仍然需要 HTTP |

这样可以同时保留三类调用方式的学习价值：

```text
外部入口：REST + Gateway
普通内部同步调用：OpenFeign
高内聚内部 RPC 调用：Dubbo
后续副作用异步调用：MQ
```

## 技术范围

| 类别 | 技术 |
|---|---|
| RPC 框架 | Dubbo |
| 注册中心 | Nacos 作为 Dubbo 注册中心 |
| 服务提供方 | `java-demo-app` |
| 服务消费方 | `task-service` |
| 接口契约 | 建议抽取公共 API/interface 模块或公共接口包 |
| 日志 | 继承 `v0.5.2` 后端运行日志基线，增加 Dubbo 调用日志 |

## 架构变化

```text
React / Vue
  -> Gateway
    -> task-service
      -> Dubbo UserRpcService -> java-demo-app
      -> OpenFeign NotificationFeignClient -> notification-service
```

REST Controller 仍然保留，用于前端、Gateway、Swagger 和 HTTP 调试。Dubbo 只作为后端内部服务调用方式。

## 建议 RPC 接口

Dubbo 接口应保持小而稳定，只暴露任务服务真实需要的用户校验能力。

| 接口 | 方法 | 说明 |
|---|---|---|
| `UserRpcService` | `getUserSummary(Long userId)` | 查询用户摘要，用于任务负责人展示或校验 |
| `UserRpcService` | `existsActiveUser(Long userId)` | 校验用户是否存在且可用 |

具体实现时可以二选一，避免为了示例暴露过多 RPC 方法。推荐优先实现 `existsActiveUser`，因为它最贴近 `task-service` 创建任务的业务需求。

## 模块与契约建议

Dubbo 依赖 Java 接口契约。建议优先考虑：

| 方案 | 说明 | 推荐度 |
|---|---|---|
| 公共 API 模块 | 新增或复用公共模块保存 Dubbo interface 和 DTO | 推荐，便于 provider/consumer 共用 |
| 复制接口 | provider 和 consumer 各自维护同名接口 | 不推荐，容易契约漂移 |
| 直接依赖服务实现模块 | consumer 依赖 provider 应用模块 | 不推荐，会破坏服务边界 |

本版本应重点学习“接口契约”和“服务边界”，不要为了省事让 `task-service` 直接依赖 `java-demo-app` 的实现代码。

## 配置要求

| 配置项 | 要求 |
|---|---|
| Dubbo application name | 每个服务使用清晰应用名 |
| Registry | 使用 Nacos 注册中心 |
| Protocol | 使用 Dubbo 默认协议或明确记录协议选择 |
| Timeout | 配置合理超时，避免任务创建被长期阻塞 |
| Retry | 谨慎配置重试，用户校验是查询类操作，可以允许有限重试 |
| Version / Group | 当前 MVP 可暂不复杂化，但文档要说明后续可用于接口演进 |
| 日志 | 记录 provider 暴露、consumer 引用、RPC 成功、RPC 失败和耗时 |

## 与 Feign 的关系

`v0.6.2` 不是为了否定 `v0.6.1`，而是为了对比两种同步调用方式。

| 调用方式 | 本项目保留位置 | 学习重点 |
|---|---|---|
| OpenFeign | `task-service -> notification-service` | HTTP、声明式客户端、复用 REST API |
| Dubbo | `task-service -> java-demo-app` 用户校验 | RPC、接口契约、强类型内部调用 |

如果实现时保留 Feign 用户校验作为临时回退，必须通过配置明确主路径和回退路径，避免同一业务链路长期混乱。

## 日志要求

继承 `v0.5.2` 后端运行日志基线。本阶段需要重点记录：

| 场景 | 建议级别 | 内容 |
|---|---|---|
| Dubbo provider 暴露 | `INFO` | 服务名、接口名、注册中心摘要 |
| Dubbo consumer 引用 | `INFO` | 调用方、接口名、目标服务 |
| RPC 调用开始 | `INFO` 或 `DEBUG` | 接口、方法、userId、requestId |
| RPC 调用成功 | `INFO` 或 `DEBUG` | userId、结果摘要、耗时 |
| RPC 调用失败 | `WARN` 或 `ERROR` | 接口、方法、异常类型、耗时、requestId |

日志仍然禁止打印完整 JWT、Authorization header、密码、数据库密码和真实密钥。

## 代码注释要求

新增和修改的后端代码必须继承当前项目注释规则：

1. Dubbo interface 需要中文 JavaDoc，说明它是内部 RPC 契约而不是对外 REST API。
2. Provider 实现需要说明如何复用用户领域逻辑，避免绕过原有业务规则。
3. Consumer 调用处需要说明为什么此链路选择 Dubbo，而通知链路仍保留 Feign。
4. 配置类和启动配置需要说明 Dubbo application、registry、protocol、timeout 的含义。
5. 方法内部关键代码块需要解释 RPC 调用、异常转换、日志记录和回退边界。

## 开发步骤

1. 确认 `v0.6.1 OpenFeign` 已完成，Nacos、Feign、日志和前端回归均可用。
2. 设计用户校验 RPC 契约，例如 `UserRpcService` 和必要 DTO。
3. 将 RPC 契约放入公共 API 模块或公共接口包，避免 `task-service` 直接依赖 `java-demo-app` 实现模块。
4. 在 `java-demo-app` 中实现 Dubbo provider，提供用户是否存在或用户摘要查询能力。
5. 在 `task-service` 中实现 Dubbo consumer，用 Dubbo 调用替代 Feign 用户校验主路径。
6. 保留 `task-service -> notification-service` 的 OpenFeign 调用，不在本版本改成 Dubbo。
7. 配置 Nacos 作为 Dubbo 注册中心，记录 Dubbo 服务注册和消费方式。
8. 增加 Dubbo 调用日志、超时配置和异常转换。
9. 补充测试，覆盖用户存在、用户不存在、RPC 调用失败等场景。
10. 回归 Gateway、React、Vue 的任务创建、任务查询和通知查询。
11. 更新 README、`docs/PROGRESS.md` 和必要决策文档。

## 验收标准

1. `java-demo-app` 可以作为 Dubbo provider 暴露用户校验 RPC 服务。
2. `task-service` 可以作为 Dubbo consumer 调用用户校验 RPC 服务。
3. 创建或更新任务时，负责人用户校验主路径通过 Dubbo 完成。
4. `task-service -> notification-service` 通知创建仍通过 OpenFeign 完成。
5. Dubbo provider 和 consumer 均注册到 Nacos 或按文档记录的注册方式可查询。
6. RPC 成功、RPC 失败、超时或 provider 不可用时有清晰日志和统一异常处理。
7. Gateway、React、Vue 的任务和通知主流程不回归。
8. 新增和修改代码已补充详细中文注释，包含方法内部关键代码块说明。

## 完成定义

2. 项目中形成清晰调用分工：外部 REST、内部 Feign、用户校验 Dubbo、通知后续 MQ。
3. Dubbo 接口契约、注册中心、超时、日志和异常处理已记录。
4. README、`docs/PROGRESS.md` 已更新。
5. 如果需要保存版本，由用户手动提交并手动打 tag：`v0.6.2-dubbo-rpc-user-validation`；Codex 不自动执行提交、tag 或 push。
