# v0.6.1 OpenFeign Service Calls

## 目标

在 `v0.6 Nacos` 已完成服务注册发现和配置中心的基础上，引入 OpenFeign，把 `task-service` 中原有手写 REST 服务调用改造为声明式 HTTP 客户端调用。

本版本只改造服务间调用方式，不新增业务接口，不改变前端操作路径，不引入 Dubbo，也不把通知链路提前改成 MQ。目标是让项目先掌握 Spring Cloud 体系下最常用的同步 HTTP 微服务调用方式。

## 新增能力

声明式服务间 HTTP 调用。

```text
task-service
  -> OpenFeign -> java-demo-app
  -> OpenFeign -> notification-service
```

## 改造范围

| 调用方 | 被调用方 | 当前目的 | v0.6.1 改造 |
|---|---|---|---|
| `task-service` | `java-demo-app` | 创建或更新任务时校验负责人用户是否存在 | 从手写 REST 调用改为 OpenFeign |
| `task-service` | `notification-service` | 任务创建、分配或状态变化后创建站内通知 | 从手写 REST 调用改为 OpenFeign |

不在本版本改造：

| 不改造项 | 原因 |
|---|---|
| 前端到 Gateway | 外部入口继续保持 REST/HTTP |
| Gateway 到后端服务 | Gateway 继续使用 HTTP 路由和 Nacos 服务发现 |
| `java-demo-app -> notification-service` | 当前不是主线必需链路，可后续按业务需要再评估 |
| Dubbo RPC | 留到 `v0.6.2` 单独学习 |
| MQ 异步通知 | 留到 `v1.0`，避免本版本同时引入消息复杂度 |

## 技术范围

| 类别 | 技术 |
|---|---|
| 声明式 HTTP 客户端 | Spring Cloud OpenFeign |
| 服务发现 | Nacos |
| 负载均衡 | Spring Cloud LoadBalancer |
| 日志 | 继承 `v0.5.2` 后端运行日志基线，增加 Feign 调用日志 |
| 认证上下文 | 继续沿用当前 JWT / requestId 透传策略 |

## 架构变化

`v0.6` 完成后，服务已经可以通过 Nacos 服务名发现。本版本把服务间调用从“自己拼 URL 并调用 REST”演进为“声明式 Feign Client”。

```text
React / Vue
  -> Gateway
    -> task-service
      -> UserFeignClient(java-demo-app)
      -> NotificationFeignClient(notification-service)
```

其中 Gateway 和前端仍然是 REST/HTTP，Feign 只用于后端服务内部的同步 HTTP 调用。

## 建议 Feign Client

| Client | 所属服务 | 目标服务 | 主要方法 |
|---|---|---|---|
| `UserFeignClient` | `task-service` | `java-demo-app` | 校验用户是否存在、查询用户摘要 |
| `NotificationFeignClient` | `task-service` | `notification-service` | 创建任务通知 |

建议只定义 `task-service` 实际需要的最小方法，不要把被调服务所有 Controller 接口都复制成 Feign Client。

## 配置要求

| 配置项 | 要求 |
|---|---|
| Feign 启用 | `task-service` 启用 Feign client 扫描 |
| 服务名 | Feign client 使用 Nacos 注册的服务名，而不是写死 `localhost:8091` 或 `localhost:8094` |
| 超时 | 配置连接超时和读取超时，避免下游服务阻塞主流程 |
| 日志级别 | 本地开发可打开 Feign 请求摘要日志，但不能打印完整 Authorization 或完整 JWT |
| requestId | Feign 调用需要继续透传 `X-Request-Id`，便于串联三个服务的日志 |
| 错误处理 | 对 4xx、5xx、超时、连接失败进行明确异常转换和日志记录 |

## 认证与安全

如果被调服务接口仍需要 JWT，则 Feign 调用需要明确处理认证上下文。

| 场景 | 建议 |
|---|---|
| 当前请求携带用户 JWT | 可以通过 Feign RequestInterceptor 透传必要认证头 |
| 内部服务调用 | 可继续沿用当前内部调用约定，后续再引入更严格的服务间认证 |
| 日志记录 | 禁止打印完整 JWT、Authorization header、密码或真实密钥 |

本版本重点是学习 OpenFeign，不建议同时重构完整服务间认证体系。

## 日志要求

继承 `v0.5.2` 后端运行日志基线。本阶段需要重点记录：

| 场景 | 建议级别 | 内容 |
|---|---|---|
| Feign 调用开始 | `INFO` 或 `DEBUG` | 被调用服务、接口、业务 ID、requestId |
| Feign 调用成功 | `INFO` 或 `DEBUG` | 状态摘要、耗时、requestId |
| Feign 调用 4xx | `WARN` | 被调服务、业务 ID、错误码、原因摘要 |
| Feign 调用 5xx / 超时 | `WARN` 或 `ERROR` | 被调服务、异常类型、耗时、是否影响主流程 |
| requestId 透传 | `DEBUG` | 只记录 requestId 是否透传，不打印敏感 header |

## 代码注释要求

新增和修改的后端代码必须继承当前项目注释规则：

1. Feign client 接口需要中文 JavaDoc，说明目标服务、调用目的和返回语义。
2. Feign 配置类、拦截器、错误解码器需要说明为什么这样配置。
3. 方法内部关键代码块需要解释 Feign 调用前校验、异常转换、requestId 透传和降级边界。
4. 不为简单 getter/setter 或普通字段赋值堆砌注释。

## 开发步骤

1. 确认 `v0.6 Nacos` 已完成，四个后端服务可以注册到 Nacos。
2. 在 `task-service` 中引入 OpenFeign 相关依赖和启用配置。
3. 定义 `UserFeignClient`，替代当前 `task-service -> java-demo-app` 的手写 REST 用户校验调用。
4. 定义 `NotificationFeignClient`，替代当前 `task-service -> notification-service` 的手写 REST 通知创建调用。
5. 增加 Feign requestId 透传能力，继续使用 `X-Request-Id` 串联日志。
6. 配置 Feign 超时、日志级别和必要错误处理。
7. 删除或收敛 `task-service` 中不再需要的手写 REST 调用封装，避免同一链路同时存在两套主实现。
8. 补充 Feign 调用相关测试，覆盖用户校验成功、用户不存在、通知创建成功、通知服务失败等场景。
9. 回归 Gateway、React、Vue 的任务创建、任务查询、通知查询和已读操作。
10. 更新 README、`docs/PROGRESS.md` 和必要决策文档。

## 验收标准

1. `task-service` 启动时可以扫描并创建 Feign client。
2. `task-service -> java-demo-app` 用户校验通过 Feign 调用完成，不再依赖写死的静态 URL。
3. `task-service -> notification-service` 通知创建通过 Feign 调用完成，不再依赖写死的静态 URL。
4. Feign 调用可以通过 Nacos 服务名发现目标服务。
5. Feign 调用可以透传 `X-Request-Id`，三个业务服务日志中能串联同一次任务创建链路。
6. 下游 4xx、5xx、超时或不可用时，有清晰日志和统一异常处理。
7. 日志中不打印完整 JWT、Authorization header、密码或真实密钥。
8. Gateway、React、Vue 的任务和通知主流程不回归。
9. 新增和修改代码已补充详细中文注释，包含方法内部关键代码块说明。

## 完成定义

1. OpenFeign 调用链路可运行、可测试、可通过真实接口验证。
2. `task-service` 中用户校验和通知创建两条同步服务调用已从手写 REST 改为 Feign。
3. Feign 相关超时、日志、requestId 透传和错误处理已记录。
4. README、`docs/PROGRESS.md` 已更新。
5. 如果需要保存版本，由用户手动提交并手动打 tag：`v0.6.1-openfeign-service-calls`；Codex 不自动执行提交、tag 或 push。
