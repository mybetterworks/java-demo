# v1.7 Observability

## 目标

引入 Prometheus、Grafana、SkyWalking 和 ELK，让系统具备指标、链路和日志三类可观测能力。

`v0.5.2` 已建立本地控制台日志和文件日志基线，本阶段要在此基础上完成日志采集、检索和 trace id 关联，而不是重新设计一套孤立日志方案。

## 新增能力

可观测性。

## 技术范围

| 类别 | 技术 |
|---|---|
| 指标 | Prometheus |
| 看板 | Grafana |
| 链路追踪 | SkyWalking |
| 日志 | ELK |

## 功能范围

| 能力 | 说明 |
|---|---|
| 指标采集 | JVM、接口耗时、请求量、错误率 |
| 看板展示 | Grafana 展示关键指标 |
| 链路追踪 | 查看 Gateway 到用户、任务、通知服务的调用链 |
| 日志检索 | 应用日志进入 Elasticsearch 并可查询 |

## 容器策略

遵守 `docs/decisions/0010-docker-service-containerization.md`。

| 服务 | 容器要求 |
|---|---|
| Prometheus | 独立容器，例如 `java-demo-prometheus-1` |
| Grafana | 独立容器，例如 `java-demo-grafana-1` |
| SkyWalking | OAP、UI 等组件分别独立容器，不与应用服务合并 |
| ELK | Elasticsearch、Logstash、Kibana 分别独立容器；如引入 Filebeat，也独立容器 |

## 开发步骤

1. 后端暴露 Actuator 指标。
2. 新增 Prometheus Docker Compose，使用独立容器。
3. 新增 Grafana Docker Compose 和基础 dashboard，使用独立容器。
4. 接入 SkyWalking Agent。
5. 配置服务名和链路采样。
6. 新增 ELK 或 Filebeat/Logstash 采集链路，各组件保持独立容器。
7. 统一日志格式，包含 trace id。
8. 复用并升级 `v0.5.2` 的本地文件日志，让 `java-demo-app`、`task-service`、`notification-service` 日志可以被采集。
9. 验证一次登录请求的指标、链路和日志。
10. 验证一次任务创建并生成通知的跨服务链路，覆盖 Gateway -> task-service -> java-demo-app -> notification-service。
11. 更新 README 和 `docs/PROGRESS.md`。

## 验收标准

1. Prometheus 可以采集应用指标。
2. Grafana 可以展示 dashboard。
3. SkyWalking 可以看到一次登录请求链路。
4. SkyWalking 可以看到一次任务创建并生成通知的跨服务链路。
5. ELK 可以检索应用日志。
6. 日志中可以关联 trace id。
7. Gateway、用户、任务、通知服务都有可观测数据。
8. `v0.5.2` 建立的本地日志文件可以被采集或明确说明迁移方式。
9. Prometheus、Grafana、SkyWalking、ELK 各组件容器边界已记录。

## 完成定义

1. 指标、链路、日志三条路径均可验证。
2. 访问地址、账号、端口已记录。
3. 本地日志、ELK 日志和 trace id 关联关系已记录。
4. 新增和修改代码已补充详细中文注释。
5. 如果需要保存版本，由用户手动提交并手动打 tag：`v1.7-observability`；Codex 不自动执行提交、tag 或 push。
