# v0.7 Redis Cache And Rate Limit

## 目标

引入 Redis，用于缓存、登录辅助能力和基础限流，为后续 Redis Cluster 做准备。

## 新增能力

缓存与限流。

## 技术范围

| 类别 | 技术 |
|---|---|
| 缓存 | Redis |
| 集群目标 | Redis Cluster |
| 网关 | Gateway 限流或业务限流 |

## 功能范围

| 功能 | 说明 |
|---|---|
| 当前用户缓存 | 缓存用户基础信息 |
| 任务列表缓存 | 缓存我的任务、任务分页或任务详情的热点查询 |
| 通知未读数缓存 | 缓存当前用户未读通知数量 |
| 登录失败计数 | 防止暴力尝试 |
| 简单限流 | 针对登录、用户查询、任务查询、通知查询接口 |
| Redis Cluster 预案 | 准备集群 compose 或说明 |

## 日志要求

继承 `v0.5.2` 后端运行日志基线。本阶段需要重点记录缓存命中、缓存未命中、缓存写入、缓存失效、限流触发、Redis 连接失败和降级策略，便于观察 Redis 对用户、任务、通知接口性能和稳定性的影响。

## 容器策略

遵守 `docs/decisions/0010-docker-service-containerization.md`。

| 阶段 | 容器要求 |
|---|---|
| 单节点验证 | 使用 Docker Desktop 拉取 Redis 镜像并运行 `java-demo-redis-1` 独立容器 |
| 集群扩展 | Redis Cluster 每个 master/replica 节点独立容器，不把多个 Redis 节点放入同一个容器 |
| 运维验证 | redis-cli、健康检查和端口说明写入 README 或 milestone 实现记录 |

## 开发步骤

1. 新增 Redis 单节点 Docker Compose，容器名、镜像版本、端口、volume 和 network 明确记录。
2. 后端接入 Redis 客户端。
3. 实现当前用户信息缓存，供用户接口和 `task-service` 用户校验链路复用。
4. 实现任务列表、任务详情或我的任务缓存，并在任务创建、更新、删除后失效。
5. 实现通知未读数缓存，并在创建通知、标记已读、全部已读后失效。
6. 实现登录失败计数和短时间锁定。
7. 在 Gateway 或后端实现基础限流，覆盖登录、用户、任务、通知关键接口。
8. 增加缓存失效逻辑，例如用户更新后删除用户缓存，任务变更后删除任务缓存，通知变更后删除未读数缓存。
9. 在缓存命中、缓存失效、限流触发和 Redis 异常降级处补充必要日志。
10. 准备 Redis Cluster compose 或单独文档，集群中每个节点对应独立容器。
11. 更新 README 和 `docs/PROGRESS.md`。

## 验收标准

1. Redis 可以启动。
2. 用户信息查询可以命中缓存。
3. 任务列表或任务详情可以命中缓存，任务变更后缓存会失效。
4. 通知未读数可以命中缓存，通知已读状态变化后缓存会失效。
5. 用户信息更新后缓存会失效。
6. 多次错误登录会触发限制。
7. 限流规则可通过接口验证。
8. 不启动 Redis 时系统有明确错误提示或降级说明。
9. 控制台和文件日志可以观察缓存命中、缓存失效、限流触发和 Redis 异常。
10. Redis 单节点和后续 Cluster 容器边界已记录。
