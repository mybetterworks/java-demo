# v0.5.2 Backend Runtime Logging

## 目标

在 `v0.5.1 Task And Notification Services` 完成 `java-demo-app`、`task-service`、`notification-service` 三个业务服务后，补齐后端运行日志能力。

本版本的重点不是引入 ELK、SkyWalking 或 Prometheus，而是在进入 `v0.5.3` 前端承接和 `v0.6 Nacos` 前先建立统一、可读、可配置、可落盘的日志基线。这样后续做前端联调、服务注册、缓存、限流、WebSocket、MQ、分布式事务和可观测性时，可以直接通过控制台和本地日志文件观察系统运行状态。

## 新增能力

后端服务运行日志基线。

同时建立后端代码阅读注释基线。后续后端开发不仅要补充类、接口、方法级中文注释，也要在方法内部的关键流程、关键分支和关键代码块前补充中文说明，帮助学习者理解“为什么这样写”和“这段逻辑在业务链路中的作用”。

```text
java-demo-app
  -> console log
  -> logs/java-demo-app.log

task-service
  -> console log
  -> logs/task-service.log

notification-service
  -> console log
  -> logs/notification-service.log
```

## 技术范围

| 类别 | 技术 |
|---|---|
| 日志门面 | SLF4J |
| 日志实现 | Spring Boot 默认 Logback |
| 控制台输出 | Spring Boot console appender |
| 文件输出 | Logback rolling file appender 或 Spring Boot logging file 配置 |
| 日志级别 | `logging.level.*`、环境变量或 profile 配置 |
| 请求标识 | 可选 MDC request id，为后续链路追踪做铺垫 |
| 代码注释 | JavaDoc、方法内关键代码块中文注释 |

## 服务范围

| 服务 | 日志目标 |
|---|---|
| `java-demo-app` | 登录、注册、JWT、用户管理、用户校验接口、异常处理 |
| `task-service` | 任务创建、任务分配、状态流转、用户服务调用、通知服务调用、异常处理 |
| `notification-service` | 通知创建、通知查询、未读数、已读标记、异常处理 |

本版本不强制改造 Gateway，避免扩大范围。若实现时发现 Gateway 请求转发日志对联调非常必要，可以只增加最小入口日志，并在实现记录中说明。

## 代码注释要求

本版本修改的是后端运行基础能力，后续阅读和排查都会频繁进入这些代码。因此新增和修改的后端代码必须补充详细中文注释，注释范围不能只停留在类和方法声明上。

| 位置 | 注释要求 |
|---|---|
| 类或配置类 | 用中文 JavaDoc 说明类职责、所属服务边界、为什么需要这个类 |
| 公开方法 | 用中文 JavaDoc 说明方法用途、关键参数、返回值、异常语义和调用场景 |
| 私有方法 | 如果承载关键业务规则、日志处理、鉴权解析、请求上下文处理或服务间调用，也需要说明职责 |
| 方法内部关键代码块 | 在关键分支、关键步骤或不容易一眼看懂的实现前增加中文块注释 |
| 配置文件 | 对日志文件路径、日志级别、pattern、profile 和环境变量覆盖方式补充中文说明 |
| 测试代码 | 说明测试覆盖的业务链路、日志场景或异常场景 |

方法内部必须重点补充注释的场景：

1. 生成、传递或清理 requestId / MDC 的代码块。
2. 登录、JWT 解析、认证失败、权限判断相关代码块。
3. 统一请求日志、响应日志、异常日志的关键处理流程。
4. 判断日志级别、日志文件路径、日志格式或 profile 的配置逻辑。
5. 调用 `java-demo-app`、`task-service`、`notification-service` 的服务间调用逻辑。
6. 任务状态流转、通知已读状态、用户状态变更等关键业务规则。
7. 异常捕获、异常转换、降级处理和返回统一错误响应的逻辑。
8. 后续版本中的缓存命中/失效、限流、消息发布/消费、事务提交/回滚等关键块。

注释风格要求：

| 要求 | 说明 |
|---|---|
| 解释意图 | 优先说明为什么这样做、当前代码解决什么问题 |
| 解释边界 | 说明当前版本为什么先做最小实现，哪些能力留给后续版本 |
| 避免废话 | 不为简单赋值、普通 getter/setter、显而易见的 if 判断重复解释 |
| 跟随代码更新 | 修改逻辑时必须同步修改附近注释，避免注释和实现不一致 |
| 不泄露敏感信息 | 注释示例中不要写真实密码、真实 token、真实密钥或真实连接串 |

## 日志文件建议

日志文件统一存储在项目运行目录下的 `logs` 目录中。该目录是运行产物，不应提交真实日志内容到 Git。

| 服务 | 建议文件 |
|---|---|
| `java-demo-app` | `logs/java-demo-app.log` |
| `task-service` | `logs/task-service.log` |
| `notification-service` | `logs/notification-service.log` |

建议支持按日期或文件大小滚动，避免本地学习时日志无限增长。早期 MVP 可以先保证控制台和单文件输出可用，随后再补充滚动策略。

## 日志格式建议

每条日志尽量包含以下信息：

| 字段 | 说明 |
|---|---|
| timestamp | 日志时间 |
| level | `TRACE`、`DEBUG`、`INFO`、`WARN`、`ERROR` |
| service | 服务名，例如 `java-demo-app` |
| thread | 当前线程 |
| requestId | 请求标识，可由过滤器生成，后续可迁移为 trace id |
| logger | 打印日志的类或模块 |
| message | 业务含义清晰的日志内容 |

示例格式：

```text
2026-05-27 20:00:00.000 INFO  [java-demo-app] [http-nio-8091-exec-1] [requestId=abc123] c.e.demo.AuthController - 用户登录成功 userId=1 username=demo
```

## 日志级别规则

| 级别 | 使用场景 |
|---|---|
| `TRACE` | 默认不启用，仅用于极细粒度排查 |
| `DEBUG` | 本地调试、关键变量摘要、服务调用参数摘要、缓存命中细节 |
| `INFO` | 服务启动、配置摘要、重要业务成功事件、关键状态变化 |
| `WARN` | 可预期异常、参数校验失败、登录失败、权限失败、下游服务可恢复失败 |
| `ERROR` | 未预期异常、下游不可用、数据写入失败、启动失败、关键配置缺失 |

默认建议：

| 环境 | 默认级别 |
|---|---|
| 本地开发 | `INFO`，按需把指定包调整为 `DEBUG` |
| 测试验证 | `INFO` |
| 故障排查 | 临时调高局部包或指定服务为 `DEBUG` |

## 配置要求

日志级别必须方便配置，不应写死在代码里。建议优先支持：

| 配置项 | 目的 |
|---|---|
| `logging.level.root` | 控制全局日志级别 |
| `logging.level.<项目包名>` | 控制本项目业务代码日志级别 |
| `logging.file.name` 或 `logging.file.path` | 控制日志文件位置 |
| 环境变量 | 本地启动时快速覆盖日志级别和文件路径 |
| profile | 区分 `local`、`test` 等配置 |

实现时可以按服务分别配置默认日志文件名，例如：

```text
java-demo-app -> logs/java-demo-app.log
task-service -> logs/task-service.log
notification-service -> logs/notification-service.log
```

## 关键日志点

### 通用日志点

| 场景 | 建议级别 | 内容 |
|---|---|---|
| 服务启动完成 | `INFO` | 服务名、端口、profile、关键依赖地址摘要 |
| 配置加载 | `INFO` | 当前激活 profile、数据库名称、下游服务地址，不打印密码 |
| 请求进入 | `INFO` 或 `DEBUG` | method、path、requestId、当前用户 ID |
| 请求结束 | `INFO` 或 `DEBUG` | method、path、status、durationMs |
| 参数校验失败 | `WARN` | 接口、字段名、失败原因摘要 |
| 业务异常 | `WARN` | 业务错误码、业务原因、requestId |
| 未预期异常 | `ERROR` | requestId、异常摘要、必要堆栈 |
| 服务间调用开始 | `INFO` 或 `DEBUG` | 调用方、被调用方、接口、requestId |
| 服务间调用结束 | `INFO` 或 `WARN` | 被调用方、状态码、耗时、失败原因 |

### java-demo-app

| 场景 | 建议级别 | 说明 |
|---|---|---|
| 注册成功 | `INFO` | 打印 userId、username，不打印密码 |
| 注册失败 | `WARN` | 用户名重复、参数不合法等 |
| 登录成功 | `INFO` | 打印 userId、username、requestId |
| 登录失败 | `WARN` | 打印 username 和失败原因摘要，不打印密码 |
| JWT 签发 | `DEBUG` | 只打印 token 类型、过期时间摘要，不打印完整 token |
| JWT 解析失败 | `WARN` | token 缺失、格式错误、过期、签名无效 |
| 用户创建/更新/删除 | `INFO` | 打印目标 userId、操作者 userId |
| 修改密码 | `INFO` | 打印目标 userId，不打印明文密码或哈希 |
| 用户校验接口 | `DEBUG` | 供 `task-service` 调用时观察校验结果 |

### task-service

| 场景 | 建议级别 | 说明 |
|---|---|---|
| 创建任务 | `INFO` | 打印 taskId、creatorUserId、assigneeUserId |
| 查询我的任务 | `DEBUG` | 打印 userId、分页参数、结果数量 |
| 修改任务状态 | `INFO` | 打印 taskId、旧状态、新状态、操作者 |
| 逻辑删除任务 | `INFO` | 打印 taskId、操作者 |
| 调用用户服务 | `INFO` 或 `DEBUG` | 打印目标用户 ID、耗时、结果 |
| 调用通知服务 | `INFO` 或 `WARN` | 打印通知创建结果，失败时记录可排查原因 |
| 任务参数异常 | `WARN` | 任务标题为空、负责人不存在、状态流转非法 |

### notification-service

| 场景 | 建议级别 | 说明 |
|---|---|---|
| 创建通知 | `INFO` | 打印 notificationId、receiverUserId、bizType、bizId |
| 查询我的通知 | `DEBUG` | 打印 userId、分页参数、结果数量 |
| 查询未读数 | `DEBUG` | 打印 userId、unreadCount |
| 标记单条已读 | `INFO` | 打印 notificationId、receiverUserId |
| 全部标记已读 | `INFO` | 打印 receiverUserId、影响数量 |
| 非本人通知操作 | `WARN` | 打印操作者 userId、notificationId，不泄露通知正文 |

## 敏感信息规则

日志必须服务于排查问题，而不是泄露数据。

禁止打印：

| 类型 | 示例 |
|---|---|
| 明文密码 | 登录密码、修改后的新密码 |
| 密码哈希 | 数据库存储的 password hash |
| 完整 JWT | `Authorization: Bearer <token>` 完整值 |
| JWT secret | 签名密钥 |
| 数据库密码 | JDBC password |
| 真实密钥 | MinIO、MQ、Redis、Nacos 等后续中间件密码 |
| 大段请求体 | 大文件、附件、长文本内容 |

允许打印经过脱敏或摘要化的信息：

| 类型 | 示例 |
|---|---|
| 用户标识 | userId、username |
| token 摘要 | 只打印 token 是否存在、过期时间、解析失败原因 |
| 下游地址 | 服务名或 host，不打印账号密码 |
| 业务 ID | taskId、notificationId、bizId |

## 开发步骤

1. 检查 `java-demo-app`、`task-service`、`notification-service` 当前日志依赖，优先使用 Spring Boot 默认 SLF4J + Logback。
2. 为三个服务分别配置控制台日志和文件日志。
3. 为三个服务分别配置默认日志文件名。
4. 增加可配置日志级别，至少支持通过 `application.yml` 或环境变量调整 root 和项目包日志级别。
5. 在三个服务的启动流程、配置加载、请求入口、业务状态变化、服务间调用和异常处理处增加必要日志。
6. 如果实现 requestId，则通过过滤器或拦截器生成并写入 MDC，响应时清理 MDC，避免线程复用导致串号。
7. 为新增和修改的后端代码补充详细中文注释，尤其是方法内部关键代码块、关键分支、服务间调用、日志上下文和异常处理逻辑。
8. 检查所有日志内容，确保不打印密码、JWT 完整 token、Authorization header、数据库密码和真实密钥。
9. 运行服务，验证控制台可以看到日志。
10. 触发登录、用户查询、任务创建、通知创建、异常场景，验证日志文件有对应记录。
11. 调整日志级别为 `DEBUG` 和 `WARN` 分别验证输出变化。
12. 更新 README 和 `docs/PROGRESS.md`，记录日志文件路径、日志级别配置方式、关键注释覆盖点和验收结果。

## 验收标准

1. `java-demo-app` 启动后控制台可以看到服务启动和关键请求日志。
2. `task-service` 启动后控制台可以看到服务启动和任务业务日志。
3. `notification-service` 启动后控制台可以看到服务启动和通知业务日志。
4. 三个服务都能把日志写入项目 `logs` 目录下的对应文件。
5. 修改日志级别为 `DEBUG` 后，可以看到更多调试日志。
6. 修改日志级别为 `WARN` 后，普通 `INFO` 和 `DEBUG` 日志不再输出。
7. 登录成功、登录失败、JWT 解析失败、任务创建、通知创建、服务间调用失败等场景都有可观察日志。
8. 日志中没有明文密码、密码哈希、完整 JWT、Authorization header、数据库密码或真实密钥。
9. 新增和修改的后端代码已有类/方法级中文注释，并在方法内部关键代码块前补充了必要中文说明。
10. 注释内容能帮助理解日志、认证、服务间调用、异常处理和关键业务流程，不只是重复代码字面含义。
11. 日志文件路径、日志级别配置方式、本地验证方式和关键注释覆盖点已记录。

## 完成定义

1. 三个业务服务都具备控制台日志和文件日志。
2. 三个业务服务的日志级别都可以方便配置。
3. 关键业务流程和异常路径都有清晰日志。
4. 敏感信息不进入日志。
5. README、`docs/PROGRESS.md` 和必要决策文档已更新。
6. 新增和修改的后端代码已补充详细中文注释，既包含类/方法说明，也包含方法内部关键代码块说明。
7. 后续后端开发必须继承本版本注释要求，新增或修改关键逻辑时同步补充方法内关键块注释。
8. 如果需要保存版本，由用户手动提交并手动打 tag：`v0.5.2-backend-runtime-logging`；Codex 不自动执行提交、tag 或 push。
