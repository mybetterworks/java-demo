# v1.6 Nginx HTTPS

## 目标

引入 Nginx 集群作为前端静态资源和网关代理入口，并配置本地 HTTPS 访问。

## 新增能力

Nginx 集群与 HTTPS 入口。

## 技术范围

| 类别 | 技术 |
|---|---|
| Web 入口 | Nginx |
| 安全访问 | HTTPS |
| 前端 | React、Vue 静态构建 |
| 后端入口 | Gateway |

## 架构目标

```text
Browser HTTPS
  -> Nginx Cluster
    -> React or Vue static files
    -> Gateway API
      -> java-demo-app
      -> task-service
      -> notification-service
```

## 容器策略

遵守 `docs/decisions/0010-docker-service-containerization.md`。

| 阶段 | 容器要求 |
|---|---|
| 单节点验证 | 使用 Docker Desktop 拉取 Nginx 镜像并运行 `java-demo-nginx-1` 独立容器 |
| 集群扩展 | Nginx 每个节点独立容器，例如 `java-demo-nginx-1`、`java-demo-nginx-2` |
| 证书和配置 | Nginx 配置、证书示例和静态资源挂载来自项目目录，不写入 Docker Desktop 全局配置 |

## 开发步骤

1. 为 React 和 Vue 增加生产构建配置。
2. 新增 Nginx 配置。
3. 配置静态资源托管。
4. 配置 `/api` 反向代理到 Gateway，由 Gateway 继续路由用户、任务和通知服务。
5. 配置 WebSocket 反向代理，重点验证通知服务实时推送链路。
6. 生成或说明本地开发证书。
7. 新增 Nginx 多节点 Docker Compose，每个 Nginx 节点独立容器。
8. 验证 HTTPS 访问。
9. 更新 README 和 `docs/PROGRESS.md`。

## 验收标准

1. React 和 Vue 可以构建为静态资源。
2. Nginx 可以托管前端页面。
3. HTTPS 可以访问前端。
4. HTTPS 页面可以调用 Gateway API，并能访问用户、任务、通知接口。
5. WebSocket 经过 Nginx 仍可用，通知推送链路不回归。
6. 多个 Nginx 节点配置一致。
7. Nginx 节点容器边界、配置挂载和证书路径已记录。
