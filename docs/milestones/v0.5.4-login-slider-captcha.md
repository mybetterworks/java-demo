# v0.5.4 Login Slider Captcha

## 状态

准备开发。

当前 `v0.5.3 Task And Notification Frontends` 已完成，下一步从本 milestone 开始实现登录风险验证。

## 目标

在 `v0.5.3 Task And Notification Frontends` 完成前端业务闭环后，为登录流程增加基础风险验证能力：

```text
同一登录主体在 5 分钟内登录失败 3 次
  -> 后续登录需要账号密码 + 滑块验证码
```

本版本重点是学习“登录风险识别 + 二次验证 + 前后端联动”的最小实现，不追求生产级验证码安全强度。验证码存储和失败计数先采用单机可运行方案，后续 `v0.7 Redis` 再迁移为 Redis TTL 计数和验证码状态存储。

## 新增能力

登录风险验证。

```text
React / Vue 登录页
  -> 用户输入账号密码
  -> 后端判断 5 分钟内失败次数
  -> 达到阈值后要求滑块验证码
  -> 账号密码 + 验证码通过后签发 JWT
```

## 功能范围

| 功能 | 说明 |
|---|---|
| 登录失败计数 | 统计 5 分钟窗口内登录失败次数 |
| 风险状态判断 | 失败达到 3 次后，后续登录必须携带滑块验证码 |
| 滑块验证码挑战 | 后端生成一次性验证码挑战，前端展示滑块 |
| 滑块验证 | 前端完成滑块后，后端校验并返回一次性验证结果或 token |
| 登录接口扩展 | 登录请求在需要验证码时同时提交账号、密码和验证码结果 |
| 成功后清理 | 登录成功后清理该登录主体的失败计数和验证码状态 |
| 前端联动 | React 和 Vue 登录页都要支持验证码提示、展示、校验和错误反馈 |

## 风险判断规则

| 规则 | 建议 |
|---|---|
| 时间窗口 | 5 分钟滚动窗口或固定 TTL 窗口 |
| 失败阈值 | 3 次 |
| 统计维度 | MVP 推荐使用 `username + clientIp`，避免单纯按账号被恶意撞库导致全局干扰 |
| 触发时机 | 第 3 次失败后进入验证码要求状态；下一次登录必须完成验证码 |
| 清理时机 | 登录成功后清理失败计数和验证码状态 |
| 过期策略 | 5 分钟无继续失败后自动恢复普通登录 |

说明：如果实现时为了简化先按 `username` 统计，也必须在文档中记录风险：攻击者可能通过不断尝试某个账号，让该账号进入验证码状态。

## 推荐接口设计

本版本可以在现有 `/api/auth/login` 基础上扩展，也可以增加验证码相关接口。

| 方法 | 路径 | 说明 | 是否需要 JWT |
|---|---|---|---|
| `POST` | `/api/auth/login` | 登录；当风险状态触发时，需要同时提交验证码结果 | 否 |
| `POST` | `/api/auth/captcha/slider` | 生成滑块验证码挑战 | 否 |
| `POST` | `/api/auth/captcha/slider/verify` | 校验滑块结果，返回一次性验证码 token 或验证通过标识 | 否 |

建议登录响应在需要验证码时返回统一错误码或业务码：

```json
{
  "code": "CAPTCHA_REQUIRED",
  "message": "登录失败次数过多，请完成滑块验证",
  "data": {
    "captchaRequired": true
  }
}
```

如果前端已经知道当前账号需要验证码，也可以先获取验证码挑战，再提交登录。

## 登录流程

### 普通登录

```text
用户输入账号密码
  -> POST /api/auth/login
    -> 未达到失败阈值
    -> 校验账号密码
    -> 成功：签发 JWT，清理失败计数
    -> 失败：记录失败次数
```

### 触发验证码后登录

```text
用户 5 分钟内失败 3 次
  -> 后端标记该登录主体需要滑块验证码
  -> 前端显示滑块验证码
  -> 用户完成滑块
  -> 后端校验滑块
  -> 用户再次提交账号密码 + 验证码结果
  -> 验证码通过且账号密码正确：签发 JWT，清理失败计数和验证码状态
```

## 滑块验证码 MVP 建议

本版本只做学习型最小实现，不追求对抗专业打码平台。

| 项目 | 建议 |
|---|---|
| challengeId | 后端生成随机 ID，用于标识一次验证码挑战 |
| 目标位置 | 后端生成目标偏移量或验证答案，设置短 TTL |
| 前端展示 | React/Vue 显示滑块组件，用户拖动后提交结果 |
| 容差 | 允许少量像素误差，例如 `±5px`，具体实现时记录 |
| 一次性 | 验证成功后 challenge 失效，避免重复使用 |
| 过期时间 | 建议 2 分钟 |
| 安全边界 | 不在日志或响应中暴露真实答案 |

如果实现图片拼图成本较高，MVP 可以先使用“滑动到指定位置”的轻量滑块；后续如果想增强安全性，再引入专业验证码组件或第三方验证码服务。

## 日志要求

继承 `v0.5.2` 后端运行日志基线。

| 场景 | 建议级别 | 日志内容 |
|---|---|---|
| 登录失败计数增加 | `WARN` 或 `INFO` | username、clientIp 摘要、失败次数、requestId |
| 触发验证码要求 | `WARN` | username、clientIp 摘要、阈值、时间窗口 |
| 生成验证码挑战 | `INFO` 或 `DEBUG` | challengeId、过期时间，不打印答案 |
| 验证码校验成功 | `INFO` | challengeId、requestId |
| 验证码校验失败 | `WARN` | challengeId、失败原因摘要 |
| 登录成功并清理状态 | `INFO` | userId、username、是否清理失败计数 |

禁止打印：

1. 明文密码。
2. 完整 JWT。
3. Authorization header。
4. 滑块验证码真实答案。
5. 可复用的验证码 token 原文。

## 数据与状态存储

本版本优先保持最小可运行。

| 状态 | MVP 建议 | 后续演进 |
|---|---|---|
| 登录失败计数 | 单机内存或轻量本地存储，记录 5 分钟 TTL | `v0.7 Redis` 使用 TTL key |
| 验证码挑战 | 单机内存保存 challengeId、答案摘要、过期时间 | `v0.7 Redis` 使用 TTL key |
| 验证通过结果 | 一次性 token 或 challenge 状态 | `v0.7 Redis` 使用短 TTL token |

如果实现时选择 MySQL 表存储，也需要记录清理策略，避免验证码和失败计数无限增长。

## 验收标准

1. 普通账号密码正确时可以正常登录并签发 JWT。
2. 5 分钟内登录失败未达到 3 次时，不强制验证码。
3. 5 分钟内登录失败达到 3 次后，后续登录必须完成滑块验证码。
4. 未完成验证码时，即使账号密码正确也不能直接登录。
5. 验证码错误或过期时，登录返回明确错误提示。
6. 验证码通过且账号密码正确时，登录成功并清理失败计数。
7. React 登录页可以展示并完成滑块验证码流程。
8. Vue 登录页可以展示并完成滑块验证码流程。
9. 日志能观察失败计数、验证码触发、验证码校验和状态清理，但不泄露密码、完整 JWT 或验证码答案。
10. 新增和修改代码已补充详细中文注释，包含方法内部关键代码块说明。
11. README 和 `docs/PROGRESS.md` 已记录本版本能力和验证结果。
