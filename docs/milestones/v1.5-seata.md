# v1.5 Seata

## 目标

引入 Seata 3 节点集群，验证跨服务分布式事务。此阶段需要服务边界足够清晰，至少涉及两个后端服务。

## 新增能力

分布式事务。

## 技术范围

| 类别 | 技术 |
|---|---|
| 分布式事务 | Seata |
| 注册配置 | Nacos |
| 数据库 | MySQL |
| 服务 | User、Task、Notification |

## 业务场景建议

优先使用任务业务验证跨服务事务边界：

1. `task-service` 写入任务表。
2. `notification-service` 写入任务分配通知。
3. 可选统计或审计服务写入任务操作记录。

任一环节失败时整体回滚。

也可以保留用户创建场景作为对照：创建用户后初始化欢迎通知或审计记录，验证用户服务和通知服务之间的一致性。

## 容器策略

遵守 `docs/decisions/0010-docker-service-containerization.md`。

| 服务 | 容器要求 |
|---|---|
| Seata | 3 个 Seata 节点分别使用 `java-demo-seata-1`、`java-demo-seata-2`、`java-demo-seata-3` 独立容器 |
| Nacos | 继续使用独立 Nacos 容器或 Nacos 集群容器，不能与 Seata 合并 |
| MySQL | Seata 元数据表和业务库使用独立 MySQL 容器或已有 MySQL 集群容器 |

## 开发步骤

1. 确认 `java-demo-app`、`task-service`、`notification-service` 的服务拆分状态是否满足 Seata 验证。
2. 新增 Seata 3 节点 Docker Compose，每个 Seata 节点独立容器。
3. Seata 接入 Nacos 注册和配置。
4. 初始化 Seata 所需数据库表。
5. 后端服务接入 Seata 客户端。
6. 实现跨服务事务业务接口，优先选择“创建任务并创建通知”的链路。
7. 添加一个可控失败开关，用于验证回滚。
8. 验证提交和回滚路径。
9. 更新 README 和 `docs/PROGRESS.md`。

## 验收标准

1. Seata 3 个节点可以启动。
2. Seata 服务可以注册到 Nacos。
3. 正常创建任务并创建通知时多个服务数据一致。
4. 制造异常时任务和通知数据一起回滚，或按文档说明选择最终一致性替代方案。
5. 事务日志和业务日志可追踪。
6. Seata、Nacos、MySQL 的容器边界和启动顺序已记录。

## 完成定义

1. 分布式事务提交可验证。
2. 分布式事务回滚可验证。
3. Seata 集群部署和故障排查已记录。
4. 新增和修改代码已补充详细中文注释。
5. 如果需要保存版本，由用户手动提交并手动打 tag：`v1.5-seata`；Codex 不自动执行提交、tag 或 push。
